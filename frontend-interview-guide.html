1.1 基础概念（必背）

Q1 说说 JS 的基本数据类型，如何判断数据类型？ 字节、腾讯高频
【参考答案】

JS 数据类型分两大类：基本类型和引用类型。

  • 基本类型（7 种）：Number、String、Boolean、null、undefined、Symbol、BigInt。存在栈内存中。
  • 引用类型：Object（含 Array、Function、Date 等）。存在堆内存中。

判断方法有四种，对应不同场景：

JavaScript
typeof 123              // 'number'   —— 判断基本类型，但 typeof null === 'object'（历史遗留 bug）
typeof [] === 'object'  // 无法区分数组和对象
[] instanceof Array     // true      —— 判断引用类型，基于原型链
Array.isArray([])       // true      —— 专门判断数组，最推荐
Object.prototype.toString.call([])  // '[object Array]'  —— 最准确，能识别所有类型
💡 记忆技巧：记 “口诀”：typeof 判基础，instanceof 判对象，toString 最通用。

Q3 == 和 === 的区别？ ★
【参考答案】

=== 是严格相等，类型和值都相同才为 true；== 是宽松相等，会进行类型转换再比较。

JavaScript
1 == '1'        // true   —— string 转 number
1 === '1'       // false  —— 类型不同
null == undefined   // true（特殊规则）
null === undefined  // false
NaN == NaN      // false（NaN 不等于任何值，包括自己）
[] == false     // true（[] 转为 0）

开发中永远用 ===，除了判断 null/undefined 时可以用 == null 一起判断。

Q4 谈谈你对闭包的理解？应用场景？ ★★
【参考答案】

闭包 = 函数 + 函数能访问的外部变量。简单说就是 “内层函数引用了外层函数的变量，哪怕外层函数已经执行完，这些变量也不会被销毁”。

JavaScript
function outer() {
  let count = 0;          // 这个变量不会被销毁
  return function inner() {
    count++;
    console.log(count);
  };
}
const fn = outer();
fn(); // 1
fn(); // 2  —— count 被保留

四个常见应用场景：

  • 数据私有化（类似 private）：通过闭包封装内部状态，外部只能通过暴露的方法访问。
  • 防抖节流：需要保存定时器 ID 等状态。
  • 柯里化：固定部分参数返回新函数。
  • React Hooks：useState 的实现本质依赖闭包。

缺点：闭包中的变量一直占用内存，过度使用会导致内存泄漏。

Q5 说说 JavaScript 的原型和原型链 ★★
【参考答案】

三个核心概念：

  • 每个构造函数有一个 prototype 属性，指向原型对象。
  • 每个实例对象有一个 __proto__ 属性，指向它构造函数的原型对象。
  • 原型对象本身也有 __proto__，指向上一级原型对象，最终指向 Object.prototype，再往上是 null —— 这条链就是原型链。

访问对象属性时，先在自身查找，找不到就沿原型链往上找，找到 null 还没找到就返回 undefined。

JavaScript
function Person() {}
const p = new Person();
p.__proto__ === Person.prototype          // true
Person.prototype.__proto__ === Object.prototype  // true
Object.prototype.__proto__ === null       // true
💡 记忆技巧：记忆：实例 → 构造函数的 prototype → Object.prototype → null，这就是原型链终点。
Q6 this 指向的判断规则？ ★★
【参考答案】

this 的指向取决于函数如何被调用，不是如何定义。四条规则按优先级从高到低：

  • new 绑定：new 调用构造函数，this 指向新创建的对象。
  • 显式绑定：call / apply / bind 指定 this。
  • 隐式绑定：obj.fn() 形式，this 指向 obj。
  • 默认绑定：直接调用 fn()，严格模式下 this 为 undefined，非严格模式下为 window。

特殊情况：

  • 箭头函数：没有自己的 this，指向定义时所在作用域的 this（词法绑定）。
  • 事件处理函数：this 指向触发事件的 DOM 元素。
JavaScript
const obj = {
  name: 'A',
  normal: function() { return this.name; },
  arrow: () => this.name
};
obj.normal();  // 'A'   —— 隐式绑定
obj.arrow();   // undefined  —— 箭头函数不绑定 this
Q7 浅拷贝和深拷贝的区别？如何实现深拷贝？ ★★
【参考答案】

浅拷贝：只复制对象的第一层，嵌套对象还是引用。深拷贝：完全独立，递归复制每一层。

常见浅拷贝方法：Object.assign()、扩展运算符 {...obj}、Array.from()。

深拷贝方法：

  • JSON.parse(JSON.stringify(obj))：简单粗暴，但无法处理 function、Symbol、循环引用、Date 等。
  • structuredClone(obj)：2022+ 浏览器原生支持，推荐。
  • lodash.cloneDeep：功能最全。

手写简易深拷贝：

JavaScript
function deepClone(obj, map = new WeakMap()) {
  if (obj === null || typeof obj !== 'object') return obj;
  if (map.has(obj)) return map.get(obj); // 处理循环引用
  const clone = Array.isArray(obj) ? [] : {};
  map.set(obj, clone);
  for (const key in obj) {
    if (obj.hasOwnProperty(key)) {
      clone[key] = deepClone(obj[key], map);
    }
  }
  return clone;
}

1.2 异步与 Event Loop（考察深度）

Q8 说说 Event Loop（事件循环） ★★★ 超高频
【参考答案】

JS 是单线程语言，Event Loop 是 JS 处理异步任务的机制。核心 4 点：

  • JS 执行时会维护一个执行栈（主线程）和任务队列。
  • 异步任务分两种：宏任务（macrotask）和微任务（microtask）。
  • 执行流程：同步代码执行完 → 清空所有微任务 → 取一个宏任务执行 → 再清空微任务 → 循环。
  • 关键：每执行完一个宏任务，都会把当前微任务队列全部清空后再执行下一个宏任务。

常见分类：

  • 宏任务：setTimeout、setInterval、setImmediate（Node）、I/O、UI Rendering、script（整体代码）。
  • 微任务：Promise.then / .catch / .finally、process.nextTick（Node，优先级更高）、MutationObserver、queueMicrotask。
JavaScript
console.log(1);
setTimeout(() => console.log(2), 0);
Promise.resolve().then(() => console.log(3));
console.log(4);
// 输出：1 4 3 2
// 解析：同步先输出 1、4 → 清空微任务输出 3 → 执行宏任务输出 2
💡 记忆技巧：记忆：微任务插队，宏任务排队。每次宏任务之后都会清空微任务。
Q9 手写一个简易 Promise ★★★ 字节、阿里必问
【参考答案】

Promise 的三种状态：pending / fulfilled / rejected，状态一旦改变不可逆。简易实现如下：

JavaScript
class MyPromise {
  constructor(executor) {
    this.status = 'pending';
    this.value = undefined;
    this.onFulfilledCallbacks = [];
    this.onRejectedCallbacks = [];
    const resolve = (value) => {
      if (this.status === 'pending') {
        this.status = 'fulfilled';
        this.value = value;
        this.onFulfilledCallbacks.forEach(fn => fn(value));
      }
    };
    const reject = (reason) => {
      if (this.status === 'pending') {
        this.status = 'rejected';
        this.value = reason;
        this.onRejectedCallbacks.forEach(fn => fn(reason));
      }
    };
    try { executor(resolve, reject); }
    catch (e) { reject(e); }
  }
  then(onFulfilled, onRejected) {
    return new MyPromise((resolve, reject) => {
      if (this.status === 'fulfilled') resolve(onFulfilled(this.value));
      else if (this.status === 'rejected') reject(onRejected(this.value));
      else {
        this.onFulfilledCallbacks.push(v => resolve(onFulfilled(v)));
        this.onRejectedCallbacks.push(v => reject(onRejected(v)));
      }
    });
  }
}

要点：状态不可逆、回调队列、then 返回新 Promise 以支持链式调用。

Q11 async/await 和 Promise 的关系？有什么优势？ ★★
【参考答案】

async/await 是 Promise 的语法糖，本质还是 Promise。

优势：

  • 代码更像同步，可读性高，告别 .then() 嵌套。
  • try/catch 可以统一处理错误，配合 async 函数更自然。
  • 调试更容易（断点打在 await 上）。

注意点：

  • await 只能在 async 函数中使用（顶层 await 是 ESM 特性）。
  • await 多个独立的请求要用 Promise.all 并发，否则会串行浪费时间。
JavaScript
// ❌ 串行，慢
const a = await fetchA();
const b = await fetchB();

// ✅ 并行，快
const [a, b] = await Promise.all([fetchA(), fetchB()]);

Q12 手写防抖和节流 ★★ 几乎必考
【参考答案】

防抖（debounce）：一段时间内只执行最后一次。应用：搜索输入、窗口 resize。

节流（throttle）：一段时间内只执行一次。应用：滚动事件、按钮防连击。

JavaScript
// 防抖
function debounce(fn, delay = 300) {
  let timer = null;
  return function(...args) {
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// 节流
function throttle(fn, delay = 300) {
  let lastTime = 0;
  return function(...args) {
    const now = Date.now();
    if (now - lastTime >= delay) {
      fn.apply(this, args);
      lastTime = now;
    }
  };
}
💡 记忆技巧：记忆方法：防抖 = 防止短时间重复触发（等它停下来）；节流 = 节省触发次数（等间隔执行）。
Q13 手写 call / apply / bind ★★★
【参考答案】
JavaScript
// call：改变 this，参数单独传
Function.prototype.myCall = function(ctx, ...args) {
  ctx = ctx || window;
  const fnKey = Symbol();
  ctx[fnKey] = this;       // this 是被调用的函数
  const res = ctx[fnKey](...args);
  delete ctx[fnKey];
  return res;
};

// apply：和 call 几乎一样，只是参数是数组
Function.prototype.myApply = function(ctx, args = []) {
  ctx = ctx || window;
  const fnKey = Symbol();
  ctx[fnKey] = this;
  const res = ctx[fnKey](...args);
  delete ctx[fnKey];
  return res;
};

// bind：返回新函数，this 永远绑定
Function.prototype.myBind = function(ctx, ...args1) {
  const fn = this;
  return function(...args2) {
    return fn.apply(ctx, [...args1, ...args2]);
  };
};

// 方法 1：递归
function flatten(arr, depth = 1) {
  return arr.reduce((acc, cur) => {
    if (Array.isArray(cur) && depth > 0) {
      return acc.concat(flatten(cur, depth - 1));
    }
    return acc.concat(cur);
  }, []);
}

Q22 Vue3 响应式原理 —— Proxy + Reflect ★★★ 超高频
【参考答案】

三个核心对象：reactive（基于 Proxy 代理对象）、ref（用 .value 包装基础类型）、effect（副作用函数，依赖收集）。

工作流程：

  • reactive 用 Proxy 代理对象，拦截 get / set 操作。
  • 组件 render 时访问响应式数据 → 触发 Proxy 的 get → 通过 track() 把当前的 effect 作为依赖收集到全局 targetMap。
  • 修改数据 → 触发 set → trigger() 从 targetMap 找到依赖的 effect 重新执行 → 组件重新渲染。
JavaScript
// 简化版
function reactive(target) {
  return new Proxy(target, {
    get(obj, key) {
      track(obj, key);          // 收集依赖
      return Reflect.get(obj, key);
    },
    set(obj, key, value) {
      const result = Reflect.set(obj, key, value);
      trigger(obj, key);        // 触发更新
      return result;
    }
  });
}
💡 记忆技巧：关键词：Proxy 代理、get 收集、set 触发、依赖是 effect。三次重复就记住了。

Q31 Vue 有了响应式系统，为什么还需要虚拟 DOM 和 diff？ ★★★ 字节爱问
【参考答案】

两个层面的原因：

  • 粒度问题：响应式系统的粒度是组件级别（整个组件重新渲染），组件内部依然可能有大量未变化的 DOM，虚拟 DOM 通过 diff 把真正要变的 DOM 定位出来。
  • 跨端能力：虚拟 DOM 是 JS 对象描述，可以跨端（Web / Native / SSR）。

Vue3 的优化：通过编译时静态分析（PatchFlags）标记动态节点，diff 只比较动态部分，跳过静态节点，性能大幅提升。

Q32 Vue3 编译时做了哪些优化？ ★★★
【参考答案】
  • 静态提升（hoistStatic）：静态节点只创建一次，跨渲染复用。
  • PatchFlags：给动态节点打上标记（1-文本、2-class、4-style、8-props...），diff 时只比较这部分。
  • cacheHandlers：事件处理函数缓存，避免每次渲染都创建新函数。
  • Block Tree：收集动态节点到 block 数组，diff 时直接线性遍历动态节点。
Q33 说说 Vue 的 nextTick 原理 ★★
【参考答案】

Vue 的 DOM 更新是异步的：同一个事件循环中的多次数据变更会合并成一次更新。nextTick 就是 “等本次更新完再执行回调” 的机制。

实现原理：内部维护一个回调队列，用微任务（Promise.then 为主，降级到 MutationObserver → setImmediate → setTimeout）在下一个 tick 执行。

JavaScript
this.list.push(1);
console.log(this.$refs.list.children.length);  // 可能是旧值
await this.$nextTick();
console.log(this.$refs.list.children.length);  // 最新值

Q46 枚举 enum 和字面量联合类型 union，2026 年该选哪个？ ★★
【参考答案】

2026 年主流方案是 const + as const 或字符串字面量联合类型，枚举用得越来越少。原因：

  • enum 编译后会生成运行时代码（一段 IIFE），增大包体积。
  • 数字枚举有类型不安全问题（任何数字都能赋值）。
  • 字面量联合编译后完全消失（零体积），且可以直接被 IDE 提示。

// ❌ 不推荐
enum Status { Pending, Success, Fail }

// ✅ 推荐：字符串字面量联合
type Status = 'pending' | 'success' | 'fail';

// ✅ 或 const 对象 + as const
const Status = { Pending: 'pending', Success: 'success', Fail: 'fail' } as const;
type Status = typeof Status[keyof typeof Status];

Q50 BFC 是什么？解决什么问题？ ★★
【参考答案】

BFC（Block Formatting Context）块级格式化上下文，是一个独立的渲染区域。

触发 BFC 的条件（任一）：

  • float 不为 none。
  • position: absolute / fixed。
  • display: inline-block / flex / grid / flow-root（最干净）。
  • overflow 不为 visible。

BFC 能解决的三大问题：

  • 外边距合并（margin collapse）：父子元素 / 兄弟元素的 margin 会合并，BFC 可阻止。
  • 清除浮动：父元素触发 BFC 可以包含内部的浮动元素。
  • 防止元素被浮动元素覆盖：与浮动元素相邻的元素触发 BFC，就不会被覆盖。

Q54 CSS 动画性能优化的关键点？ ★★
【参考答案】
  • 用 transform 和 opacity 做动画 —— 只触发合成层（Composite），不触发重排/重绘。

Q55 CSS 重排（Reflow）和重绘（Repaint）的区别？ ★★
【参考答案】

重排：改变布局（宽高、位置、显示隐藏）会重新计算 DOM 几何属性，代价高。

重绘：只改变外观（背景色、颜色），不改变布局，代价低。

重排一定触发重绘，重绘不一定触发重排。

触发重排的操作：

  • 读取布局信息：offsetWidth、clientHeight、getBoundingClientRect。
  • 改变尺寸/位置：width、margin、padding、top/left。
  • 改变字体、内容、添加/删除 DOM。
💡 记忆技巧：优化：批量修改 DOM（先设 display:none 修改后再显示）、用 transform 代替 top/left、读写分离（防止强制同步布局）。

Q59 Webpack 的构建流程？ ★★ 必会

Q62 Webpack 的 Tree Shaking 原理？ ★★
【参考答案】

Tree Shaking = 摇树，通过 ES Module 的静态结构特性，在打包时移除未使用的代码。

三个前提条件：

  • 必须使用 ES Module（import/export），CommonJS 的 require 不行。
  • mode: 'production'（自动开启 UglifyJs / Terser 删除无用代码）。
  • package.json 设置 "sideEffects": false 或列出有副作用的文件。

sideEffects 告诉打包器：这个包的模块没有副作用（没有只执行代码，没有 import 'polyfill' 这种），可以安全删除未使用的导出。

Q63 HMR（热模块替换）原理？ ★★
【参考答案】

流程：

  • webpack-dev-server 启动时，通过 WebSocket 和浏览器建立长连接。
  • 文件改动 → Webpack 重新编译 → 生成新的 manifest.json 和 hotUpdate 文件。
  • 服务端通过 WebSocket 推送新 hash 给浏览器。
  • 浏览器端 HMR Runtime 通过 JSONP 拉取更新文件，替换变化的模块，保留状态。

这就是为什么改 Vue 组件不会刷新整个页面、输入框里的内容不会丢失。

4.2 Vite 原理

Q64 Vite 为什么比 Webpack 快？ ★★★ 字节高频
【参考答案】

从两个阶段分析：

① 开发阶段（冷启动、热更新）：

  • Webpack：必须先打包整个项目再启动，项目越大越慢。
  • Vite：基于浏览器原生 ES Module，启动时不打包，浏览器发请求才按需编译当前模块。
  • 依赖预构建：Vite 用 esbuild 把 node_modules 依赖预打包成 ESM，esbuild 用 Go 写，比 JS 快 10-100 倍。
  • HMR：Vite 只需要让浏览器重新请求变化的模块，热更新速度不受项目大小影响。

② 生产阶段：

  • Vite 生产环境用 Rollup 打包（浏览器加载大量 ESM 请求开销大）。
  • Rollup 的 Tree Shaking 比 Webpack 更纯粹。
💡 记忆技巧：口诀记忆：Webpack 先打包后启动，Vite 边用边编译；esbuild 预构建 + ESM 按需加载 = 快。
Q65 Vite 的预构建是什么？为什么需要？ ★★
【参考答案】

Vite dev server 启动时，会扫描依赖，把 CommonJS / UMD 模块（如 react、lodash）用 esbuild 预打包成 ESM，缓存在 node_modules/.vite。

为什么需要：

  • 很多 npm 包还不是纯 ESM，浏览器不认识，需要转换。
  • 一个包可能有大量内部模块（lodash 有 600+ 小文件），浏览器按需加载会发出大量请求，慢。预构建合并成单个 ESM 文件后，只需一个请求。
Q66 Vite 的 HMR 和 Webpack 的 HMR 有什么不同？ ★★
【参考答案】
  • Webpack：文件变化 → 重新编译受影响的模块链 → WebSocket 通知浏览器 → 替换模块。
  • Vite：文件变化 → 只需让浏览器重新请求这个模块（ESM 的好处）。
  • 速度差异：Webpack 会随着项目增大变慢（编译依赖链长），Vite 的 HMR 速度几乎恒定。
Q67 Vite 和 Webpack 如何选择？ ★★
【参考答案】
  • 新项目、现代浏览器、Vue3 / React18+ → Vite（开发体验好）。
  • 复杂老项目、需要兼容 IE、微前端老架构 → Webpack（生态成熟）。
  • 库打包（组件库、npm 包）→ Rollup 或 Vite 的库模式。

5.1 浏览器原理

Q71 从输入 URL 到页面呈现的全过程？ ★★★ 大厂必问
【参考答案】

分八步，按先后顺序讲：

  • ① URL 解析：判断是合法 URL 还是搜索关键词。
  • ② DNS 查询：浏览器缓存 → OS 缓存 → 路由器 → DNS 服务器，找到 IP。
  • ③ TCP 三次握手：建立连接。HTTPS 还需要 TLS 握手。
  • ④ 发送 HTTP 请求：浏览器构造请求报文发给服务器。
  • ⑤ 服务器响应：返回 HTML 等资源。
  • ⑥ 浏览器解析：HTML 构建 DOM 树，CSS 构建 CSSOM 树，合成 Render 树。
  • ⑦ 布局 + 绘制：计算几何位置（Layout）、绘制像素（Paint）、合成图层（Composite）。
  • ⑧ JS 执行：遇到 script 标签阻塞渲染（除非 async/defer），执行完继续渲染。
💡 记忆技巧：关键扩展：可以补充协商缓存、强缓存触发时机；HTTP2 多路复用；关键渲染路径优化。
Q72 浏览器的缓存机制？强缓存和协商缓存的区别？ ★★ 必问
【参考答案】

两级缓存：浏览器先查强缓存，强缓存失效后再走协商缓存。

对比项强缓存协商缓存
是否发请求不发发（询问服务器）
状态码200 (from cache)304 Not Modified
响应头Cache-Control（优先）、ExpiresLast-Modified / If-Modified-Since，ETag / If-None-Match
用途缓存期内直接用本地过期后问服务器有没有更新

为什么 Cache-Control 优先级比 Expires 高：Expires 是绝对时间，客户端时间不准会出问题；Cache-Control 是相对时间（max-age=3600 秒），更可靠。

为什么需要 ETag：Last-Modified 精度到秒，同一秒内的修改检测不到；文件内容不变但时间戳改变也会失效。ETag 是内容 hash，更精确。

Q73 浏览器渲染流程？什么是回流和重绘？ ★★
【参考答案】

渲染 5 步：解析 HTML → DOM 树；解析 CSS → CSSOM 树；合成 Render 树；Layout 布局；Paint 绘制。

回流（Reflow）：改变布局引起的重新计算，如修改宽高、位置、display。

重绘（Repaint）：外观变化但不影响布局，如 background-color、visibility。

合成（Composite）：GPU 层面的移动/透明度变化，不触发回流重绘（transform / opacity）。

性能：合成 > 重绘 > 回流，动画尽量用 transform / opacity。

Q77 TCP 三次握手和四次挥手？ ★★
【参考答案】

三次握手（建立连接）：

  • ① 客户端 → 服务器：SYN=1, seq=x
  • ② 服务器 → 客户端：SYN=1, ACK=1, seq=y, ack=x+1
  • ③ 客户端 → 服务器：ACK=1, seq=x+1, ack=y+1

为什么不是两次：防止失效的连接请求到达服务器后建立无效连接。

四次挥手（断开连接）：

  • ① 客户端 → 服务器：FIN，我要断开。
  • ② 服务器 → 客户端：ACK，我知道了，等我处理完。
  • ③ 服务器 → 客户端：FIN，我处理完了，可以断开。
  • ④ 客户端 → 服务器：ACK，收到，彻底断开。

为什么比握手多一次：服务器收到 FIN 后不能立刻关闭（还有数据要发），所以 ACK 和 FIN 分两次发。

Q80 JWT 认证机制的原理和优缺点？ ★★
【参考答案】

JWT = Header.Payload.Signature 三段 base64 字符串。服务器签发，客户端保存，后续请求放在 Authorization: Bearer xxx 头。

优点：

  • 无状态：服务器不用保存 session，易于横向扩展。
  • 跨域友好：放在 header 里，不依赖 cookie。
  • 自包含：token 里直接带用户信息，减少数据库查询。

缺点：

  • 无法主动失效：签发后到过期前一直有效（可以通过服务器黑名单解决）。
  • 体积大：比 session id 大。
  • Payload 可读（只是 base64 编码不是加密），不能放敏感信息。

进阶：双 Token 机制（access token 短期 + refresh token 长期），拦截器无感刷新。

5.3 前端安全

Q81 XSS 攻击和防御 ★★
【参考答案】

XSS（跨站脚本攻击）：攻击者向页面注入恶意脚本，在其他用户浏览器中执行。

三种类型：

  • 反射型：恶意代码在 URL 参数中，点击链接触发（URL → 后端 → 返回页面 → 执行）。
  • 存储型（危害最大）：恶意代码被存入数据库，所有访问的用户都受害（如评论区 <script>）。
  • DOM 型：前端 JS 直接操作 DOM 引发，不经过后端。

防御：

  • 输入过滤 + 输出转义（把 < > 转为 &lt; &gt;）。
  • HttpOnly Cookie：阻止 JS 读取，防止 token 被盗。
  • CSP（Content-Security-Policy）：白名单限制脚本来源。
  • 避免 innerHTML、eval、new Function 等危险 API。
  • Vue / React 模板自动转义（{{ }} 是安全的，v-html / dangerouslySetInnerHTML 是危险的）。
Q82 CSRF 攻击和防御 ★★
【参考答案】

CSRF（跨站请求伪造）：诱导用户在已登录 A 网站的情况下访问恶意网站 B，B 站伪造 A 站的请求（利用浏览器自动携带 Cookie）。

防御：

  • CSRF Token：后端生成随机 token，前端每次请求携带（不在 cookie 里），后端校验。
  • SameSite Cookie：Strict 或 Lax，阻止跨站请求自动带 cookie。
  • Referer / Origin 校验：后端判断来源域。
  • 关键操作二次验证（密码、验证码）。
💡 记忆技巧：口诀：XSS 偷 Cookie（盗用身份）；CSRF 借 Cookie（伪造操作）。两个同时防。
Q83 点击劫持（ClickJacking）和防御 ★
【参考答案】

攻击方式：用透明 iframe 嵌入目标网站，诱导用户点击。

防御：

  • 响应头 X-Frame-Options: DENY / SAMEORIGIN（旧方案）。
  • CSP 指令 frame-ancestors（新方案）。
  • JS 判断：if (top !== self) top.location = self.location。

Q84 核心 Web 性能指标（Core Web Vitals）是什么？ ★★
【参考答案】

Google 定义的三大核心指标（2024 起 INP 替换了 FID）：

  • LCP（Largest Contentful Paint）最大内容绘制：页面主要内容加载时间。目标 < 2.5s。
  • INP（Interaction to Next Paint）下次交互绘制：用户交互响应性。目标 < 200ms。
  • CLS（Cumulative Layout Shift）累计布局偏移：视觉稳定性。目标 < 0.1。

其他常关注：

  • FCP（First Contentful Paint）首次内容绘制。
  • TTFB（Time to First Byte）首字节时间。
  • FP（First Paint）首次绘制。

Q87 虚拟列表（Virtual List）的实现原理？ ★★★
【参考答案】

问题：10 万条数据全部渲染到 DOM，浏览器卡死。

实现要点：

  • 外层容器固定高度 + overflow: auto，内层用 padding-top / bottom 撑开总高度。
  • 监听 scroll，计算当前滚动位置对应的 startIndex / endIndex。
  • 只渲染 [startIndex, endIndex] 范围内的数据。

// 简化版伪代码
const itemHeight = 50;
const visibleCount = Math.ceil(containerHeight / itemHeight);
const onScroll = (e) => {
  const scrollTop = e.target.scrollTop;
  const startIndex = Math.floor(scrollTop / itemHeight);
  const endIndex = startIndex + visibleCount + buffer;
  visibleData = allData.slice(startIndex, endIndex);
  offsetY = startIndex * itemHeight;  // 顶部偏移
};

Q98 ECharts 在 Vue 项目中如何封装？ ★★
【参考答案】

封装目标：按需引入减小体积、自动 resize、生命周期自动销毁。

<template>
  <div ref="chartRef" :style="{ width, height }"></div>
</template>

<script setup>
import { ref, onMounted, onBeforeUnmount, watch, shallowRef } from 'vue';
import * as echarts from 'echarts/core';
import { LineChart, BarChart } from 'echarts/charts';
import { GridComponent, TooltipComponent } from 'echarts/components';
import { CanvasRenderer } from 'echarts/renderers';

echarts.use([LineChart, BarChart, GridComponent, TooltipComponent, CanvasRenderer]);

const props = defineProps({
  option: Object,
  width: { type: String, default: '100%' },
  height: { type: String, default: '400px' }
});

const chartRef = ref(null);
const chart = shallowRef(null);  // ⚠️ 用 shallowRef，避免 echarts 实例被响应式

const resize = () => chart.value?.resize();

onMounted(() => {
  chart.value = echarts.init(chartRef.value);
  chart.value.setOption(props.option);
  window.addEventListener('resize', resize);
});

watch(() => props.option, (val) => {
  chart.value?.setOption(val, true);
}, { deep: true });

onBeforeUnmount(() => {
  window.removeEventListener('resize', resize);
  chart.value?.dispose();
});
</script>

Q99 大屏项目自适应方案？ ★★
【参考答案】

大屏（1920×1080 设计稿）在不同分辨率屏幕上的自适应，三种主流方案：

  • ① rem + postcss-px-to-rem：根据屏宽动态计算 html 字号，所有 px 转 rem。移动端常用。
  • ② vw/vh：px 转视口单位。
  • ③ transform: scale（推荐大屏）：设计稿 1920×1080 按比例缩放，保持布局完全不变形。
css
/* scale 方案核心 */
.screen {
  width: 1920px;
  height: 1080px;
  transform-origin: top left;
  position: absolute;
  top: 50%;
  left: 50%;
}

// JS 动态计算 scale
function resize() {
  const scaleX = window.innerWidth / 1920;
  const scaleY = window.innerHeight / 1080;
  const scale = Math.min(scaleX, scaleY);
  screen.style.transform = `translate(-50%, -50%) scale(${scale})`;
}

Q104 预警围栏判定算法的实现思路？ ★★★
【参考答案】

这是你水上预警项目的核心亮点。核心算法：判断点是否在多边形内。

常用算法：射线法（Ray Casting）

  • 从目标点向任意方向（通常向右）发射一条射线。
  • 计算该射线与多边形各边的交点数。
  • 交点数为奇数 → 在内部；为偶数 → 在外部。
JavaScript
// 点是否在多边形内（经纬度数组）
function pointInPolygon(point, polygon) {
  const [x, y] = point;
  let inside = false;
  for (let i = 0, j = polygon.length - 1; i < polygon.length; j = i++) {
    const [xi, yi] = polygon[i];
    const [xj, yj] = polygon[j];
    const intersect = ((yi > y) !== (yj > y)) &&
      (x < (xj - xi) * (y - yi) / (yj - yi) + xi);
    if (intersect) inside = !inside;
  }
  return inside;
}

生产环境优化：

  • OpenLayers / Turf.js 有现成方法 booleanPointInPolygon（底层 RBush 空间索引）。
  • 船只大量时，用 R-Tree 空间索引快速剪枝：先判断点在哪个围栏的 bbox 内。
  • 后端地理数据库（PostGIS）可以用 ST_Contains 做服务端判断。

7.1 React 核心概念

Q111 React 的核心思想？和 Vue 的根本区别？ ★★ 跨框架题必问
【参考答案】

React 核心思想：UI = f(state)。状态驱动视图，一切都是组件。

和 Vue 的根本区别：

  • 思维模式：Vue 是 “数据变化会自动追踪”（响应式）；React 是 “每次 state 变化就整个组件重新执行”。
  • 模板：Vue 用模板 + 指令；React 用 JSX（JS 写 UI）。
  • 数据流：Vue 双向绑定（v-model 糖）；React 单向数据流（父传子通过 props，子改父通过回调）。
  • 更新粒度：Vue3 靠 Proxy 精确知道哪个数据变了；React 整棵子树重新渲染（通过 memo 手动优化）。
💡 记忆技巧：面试说法：“Vue 是框架，约束更多；React 更像库，自由度更高。两者没有绝对优劣，Vue 工程化开箱即用，React 生态需要自己选型。”
Q112 JSX 是什么？如何工作？ ★
【参考答案】

JSX 是 JS 的语法扩展，长得像 HTML，编译后变成 React.createElement 调用。

JavaScript
// JSX
const el = <h1 className="title">Hello</h1>;

// 编译后
const el = React.createElement('h1', { className: 'title' }, 'Hello');

// React 17+ 新 JSX 转换后

注意点：class → className、for → htmlFor、事件用驼峰 onClick。

Q113 函数组件 vs 类组件？为什么 Hooks 出现？ ★★
【参考答案】

类组件问题：this 绑定困扰、生命周期逻辑分散、状态逻辑复用只能靠 HOC / render props（结构嵌套多）。

函数组件 + Hooks 的优势：

  • 函数式更简洁，没有 this。
  • 逻辑按功能组织（而不是按生命周期分散）。
  • 自定义 Hook 轻松复用状态逻辑。
  • 更好的性能（函数组件开销小）。

2026 年：除了少量历史项目，新写组件全部函数组件 + Hooks。

7.2 Hooks 核心

Q114 useState 的使用和原理 ★★
【参考答案】
JavaScript
const [count, setCount] = useState(0);

// 异步更新，以下写法有问题：
setCount(count + 1);
setCount(count + 1);  // 两次都是 count+1，最终只加 1

// 正确：函数式更新
setCount(prev => prev + 1);
setCount(prev => prev + 1);  // 最终 +2

// 懒初始化（只有首次渲染执行）
const [data, setData] = useState(() => expensiveCalc());

原理：React 内部用链表维护每个组件的 hooks 状态，按调用顺序对应。所以 Hooks 不能在循环 / 条件 / 嵌套中使用。

Q115 useEffect 的用法和陷阱 ★★★ 必问
【参考答案】
JavaScript
// 1. 组件挂载 + 每次更新都执行
useEffect(() => { /* ... */ });

// 2. 仅挂载时执行一次（相当于 componentDidMount）
useEffect(() => { /* ... */ }, []);

// 3. 依赖变化时执行
useEffect(() => { /* ... */ }, [userId]);

// 4. 清理副作用（相当于 componentWillUnmount）
useEffect(() => {
  const timer = setInterval(...);
  return () => clearInterval(timer);  // 清理函数
}, []);

常见陷阱：

  • 依赖数组不完整：ESLint 会警告 exhaustive-deps，少写依赖可能导致闭包陷阱（拿到旧值）。
  • 依赖是对象/数组：每次渲染新建引用导致 effect 反复执行 —— 用 useMemo 缓存依赖。
  • 在 effect 中 setState 但没有依赖控制 → 无限循环。
  • React 18 严格模式下 effect 会执行两次（检测副作用），要保证清理函数正确。
Q116 useMemo 和 useCallback 的区别？ ★★
【参考答案】

两者都用于性能优化，都接收 fn 和依赖数组。

  • useMemo(fn, deps)：缓存 fn 的执行结果（值）。
  • useCallback(fn, deps)：缓存 fn 本身（函数引用）。
JavaScript
// useMemo 缓存昂贵计算
const doubled = useMemo(() => count * 2, [count]);

// useCallback 缓存函数引用（传给 memo 子组件避免重渲染）
const handleClick = useCallback(() => {
  console.log(count);
}, [count]);

// 等价关系
useCallback(fn, deps) === useMemo(() => fn, deps);
💡 记忆技巧：使用原则：不要无脑包 useMemo/useCallback。大多数场景开销比收益大。只在子组件是 memo 且依赖此函数、或计算真的昂贵时才用。
Q117 useRef 有哪些用法？ ★★
【参考答案】
  • 获取 DOM 引用：<input ref={inputRef} />
  • 保存可变值（不触发重渲染）：ref.current = anything。
  • 保存前一次的 state / props。
  • 保存定时器 ID、WebSocket 实例等。
JavaScript
// 保存上一次 props
function usePrevious(value) {
  const ref = useRef();
  useEffect(() => { ref.current = value; });

}

Q119 自定义 Hook 怎么写？举一个实际例子 ★★★
【参考答案】

自定义 Hook 就是一个以 use 开头的函数，内部可以用其他 Hook。作用：复用状态逻辑。

JavaScript
// useRequest - 封装请求状态
function useRequest(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    setLoading(true);
    fetch(url, { signal: controller.signal })
      .then(res => res.json())
      .then(setData)
      .catch(err => {
        if (err.name !== 'AbortError') setError(err);
      })
      .finally(() => setLoading(false));

    return () => controller.abort();  // 清理
  }, [url]);

  return { data, loading, error };
}

// 使用
function UserList() {
  const { data, loading, error } = useRequest('/api/users');
  if (loading) return <div>Loading...</div>;
  if (error) return <div>{error.message}</div>;
  return <ul>{data.map(u => <li>{u.name}</li>)}</ul>;
}

Q120 React 的虚拟 DOM 和 diff 算法 ★★
【参考答案】

Virtual DOM：用 JS 对象描述 DOM 结构，变化时先在虚拟 DOM 比较，计算最小变更再应用到真实 DOM。

React diff 的三个策略：

  • Tree diff：只对比同层节点，跨层级的移动会销毁重建。
  • Component diff：同类型组件复用实例；不同类型销毁重建。
  • Element diff：同级子节点用 key 区分，没有 key 会按 index 对比。

Q121 React Fiber 架构是什么？ ★★★
【参考答案】

React 16 前：递归 diff 整棵树，过程不可打断，长任务会阻塞主线程导致卡顿。

Fiber：把渲染工作拆成小单元，可以中断、恢复、优先级调度。

核心机制：

  • Fiber 节点：每个组件对应一个 Fiber 节点，存状态、优先级、指向父/子/兄弟。
  • 双缓存：current Fiber 树（当前屏幕显示）+ workInProgress 树（正在构建），完成后切换。
  • 时间切片：借助浏览器空闲时间（requestIdleCallback 思想）执行渲染，每个任务不超过 5ms。
  • 优先级：用户输入 > 动画 > 数据请求，高优先级打断低优先级。

React 18 的 concurrent 特性（useTransition、useDeferredValue）基于 Fiber。

Q131 Python 的装饰器是什么？ ★★
【参考答案】

装饰器本质是一个高阶函数：接收一个函数，返回一个增强版的函数。语法糖 @decorator。

Python
import time

def timer(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f'{func.__name__} took {time.time()-start:.2f}s')
        return result
    return wrapper

@timer
def slow_func():
    time.sleep(1)

slow_func()  # 输出：slow_func took 1.00s

Q132 Python 的 GIL 是什么？对性能有什么影响？ ★★
【参考答案】

GIL（Global Interpreter Lock）全局解释器锁：CPython 为了内存管理安全，同一时刻只允许一个线程执行 Python 字节码。

  • CPU 密集型任务：多线程无加速（反而因锁切换变慢），应用多进程（multiprocessing）。
  • I/O 密集型任务：多线程仍有效（I/O 时会释放 GIL）。
  • 协程（asyncio）：单线程高并发 I/O 的最佳方案，FastAPI 就是基于 asyncio。
  • 3.13 开始有实验性 no-GIL 模式（--disable-gil），但生产未普及。

class UserCreate(BaseModel):
    name: str = Field(..., min_length=2, max_length=20)

        return v

11.1 LLM 基础认知

Q164 什么是 LLM？它和传统 NLP 模型的本质区别？ ★ 开场题
【参考答案】

LLM（Large Language Model）基于 Transformer 架构，在海量文本上预训练得到的通用语言模型。

和传统 NLP 的三大本质区别：

  • 范式转变：传统 NLP 一个任务训一个模型（情感分析、命名实体识别各自为战）；LLM 一个模型做所有任务，靠 Prompt 区分。
  • 涌现能力：参数规模越过阈值（约 100 亿）后，出现小模型没有的能力（推理、翻译、代码生成）。
  • 零样本 / 少样本学习：不需要为每个任务标注大量数据，给几个例子就能做。
Q165 Transformer 的自注意力机制？为什么比 RNN 更适合处理长序列？ ★★
【参考答案】

自注意力（Self-Attention）核心公式：Attention(Q,K,V) = softmax(QK^T / √d) · V。

工作流程：

  • 每个 token 生成 Query、Key、Value 三个向量。
  • Query 和所有 token 的 Key 计算相似度，得到注意力权重。
  • 用权重对 Value 加权求和，得到该位置的新表示。

比 RNN 的三大优势：

  • 并行计算：RNN 要按时间步顺序处理，自注意力可以并行处理整个序列。
  • 长距离依赖：RNN 远距离信息会衰减，自注意力任意两个位置都是 O(1) 距离。
  • 可解释性：注意力权重可视化能看到模型关注了哪些 token。
💡 记忆技巧：缺点：计算复杂度 O(n²)，序列长度受限。解决方案包括 Flash Attention、稀疏注意力、RoPE 位置编码。
Q166 Temperature、Top-p、Top-k 采样参数的区别？ ★★ 调参必问
【参考答案】
参数作用取值建议

Q167 幻觉（Hallucination）是什么？怎么缓解？ ★★★
【参考答案】

幻觉指 LLM 生成的内容看起来流畅合理，但事实错误或凭空捏造。

产生原因：

  • 训练数据有噪声或过时。
  • 模型预测的本质是概率最大化，不是事实对齐。
  • 长上下文时中间信息容易被忽略（Lost in the Middle 现象）。

缓解方案：

  • RAG：把权威知识库接入，让模型基于检索内容回答。
  • Chain-of-Thought：让模型先推理再给答案，减少臆测。
  • Self-Consistency：采样多次取一致答案。
  • 引用溯源：强制模型标注信息来源。
  • 事实核查 Agent：生成后用另一个 Agent 验证关键事实。
  • 微调：领域数据微调让模型更贴近事实。
  • temperature=0 + 约束生成（JSON Schema、正则）。

从基础到进阶 6 层技巧：

  • ① Zero-shot：直接问。适合简单任务。
  • ② Few-shot：给 2-5 个例子（In-Context Learning）。
  • ③ Chain-of-Thought（CoT）：让模型 "一步步思考"，提升推理能力。

11.2 Agent 架构与规划

Q169 什么是 AI Agent？它和普通 LLM 调用的本质区别？ ★★★ 必问
【参考答案】

普通 LLM 调用：输入 prompt → 输出文本。一次对话一次响应，被动。

Agent：让 LLM 具备自主规划任务、调用工具、处理复杂流程的能力。主动。

Agent 的四大核心模块（面试标准答案）：

  • ① Profile（角色）：定义 Agent 是谁、能做什么、行为准则。
  • ② Memory（记忆）：短期记忆（当前对话上下文）+ 长期记忆（历史对话、知识库）。
  • ③ Planning（规划）：任务分解、工具选择、路径规划，遇到问题能反思调整。
  • ④ Action（行动）：调用工具、API、执行代码，影响外部世界。
💡 记忆技巧：一句话：Agent = LLM + Memory + Tools + Planning。
Q170 ReAct、CoT、ToT 三种规划方法的区别？ ★★★ 2026 高频
【参考答案】
模式核心思路适用场景
CoT（思维链）让 LLM 一步步推理再答数学推理、逻辑题
ReAct（Reason+Act）交替推理和调用工具Agent 工具调用主流模式
ToT（思维树）树形搜索多条推理路径可回溯复杂问题、需要探索
Reflexion（反思）执行后自我反思，失败重试长任务、容错场景
Plan-and-Execute先规划完整计划，再逐步执行多步骤任务

ReAct 是目前 Agent 应用最常用的模式，典型流程：

ReAct Pattern
Thought: 我需要先搜索 2026 年 AI 趋势相关文章
Action: web_search("AI trends 2026")
Observation: [搜索结果...]
Thought: 我已经有足够信息，可以总结回答了
Action: finish("根据搜索结果，2026 年 AI 三大趋势是...")
Q171 Agent 常见失败模式？如何解决？ ★★★ 实战题
【参考答案】

4 大典型失败场景（面试官爱追问的重点）：

① 工具调用出错：

  • 现象：LLM 生成的 JSON 参数格式不对，调用工具失败。
  • 解法：参数校验层 + 格式错误让 LLM 重新生成 + 关键调用加人工兜底。

② 上下文溢出：

  • 现象：对话轮数多了，Context 超上限，Agent 忘了之前任务。
  • 解法：上下文压缩、关键信息提取、定期 summarize、sliding window、向量检索历史消息。

③ 目标漂移：

  • 现象：Agent 越跑越偏离原始目标，在小问题里绕圈。
  • 解法：每一步做目标对齐、定期反思总结、必要时重新规划、设置最大步数兜底。

④ 无限循环：

  • 现象：同样的 Thought 和 Action 反复执行。
  • 解法：检测重复状态、step limit、引入人工介入点（Human-in-the-Loop）。
Q172 如何设计一个生产级的 Agent 框架？ ★★★ 架构题
【参考答案】

从六个维度设计（这套框架能答出深度）：

  • ① 编排层：用有向图（DAG）或状态机组织节点，每个节点是一个能力单元。LangGraph 就是这思路。
  • ② 工具层：标准化工具接口（类 OpenAI Function Calling），描述用 JSON Schema，支持权限/超时/重试。
  • ③ 记忆层：短期（对话上下文）+ 长期（向量库/KG）+ 结构化（数据库）三层记忆协同。
  • ④ 可观测层：每次推理、工具调用、状态转换全量记录，LangSmith 这类平台支持回放和 debug。
  • ⑤ 安全层：输入输出 Guardrails、敏感信息脱敏、工具权限最小化原则。
  • ⑥ 评估层：离线 benchmark + 线上 A/B + 人工抽检，数据回流迭代。
Q173 多 Agent 协作的优势和挑战？ ★★
【参考答案】

优势：

  • 分工明确：不同 Agent 负责不同领域（规划 Agent、执行 Agent、审核 Agent）。
  • 上下文独立：每个 Agent 自己的 context，不互相污染。
  • 并行提效：独立子任务可以并发执行。
  • 可插拔：某个 Agent 可以独立升级替换。

挑战：

  • 通信开销：Agent 间传递信息的 token 成本。
  • 状态同步：多个 Agent 共享状态的一致性问题。
  • 错误传播：一个 Agent 失败可能影响整条链。
  • 调度复杂度：谁先谁后、谁主谁辅。
  • 成本失控：多 Agent 串联调用，单次任务可能烧十几美元。

典型架构：Supervisor 主管模式 或 Peer-to-Peer 对等模式（A2A 协议正在标准化后者）。

【参考答案】

RAG 流程（离线+在线两阶段）：

【离线】数据准备阶段：

  • ① 文档加载：PDF / Word / HTML / Markdown → 统一 Document 对象。
  • ② 文档切分：RecursiveCharacterTextSplitter，优先在段落/句子边界切，chunk_size 500-1000。
  • ③ 向量化：Embedding 模型把文本转向量（OpenAI text-embedding-3-large、BGE、M3E）。
  • ④ 存储：向量 + 元数据存入向量数据库（Qdrant / Milvus / Chroma）。

【在线】问答阶段：

  • ⑤ 问题向量化：用户 query 也转向量。
  • ⑥ 相似度检索：从向量库取 top-k 个最相关文档块。
  • ⑦ 重排序（可选）：用 Rerank 模型精选 top-3。
  • ⑧ 上下文拼接：把检索结果注入 Prompt。
  • ⑨ 生成答案：LLM 基于上下文生成回答。

相比微调的优势：

  • 更新快：换个文档就行，不用重训练。
  • 成本低：不需要 GPU 训练。
  • 可溯源：答案可以给出引用文档。
  • 知识广：不受模型参数容量限制。
💡 记忆技巧：选择：知识频繁更新、需要溯源 → RAG；改变模型风格、深度领域能力 → 微调；两者可以结合。
Q175 RAG 检索质量差？怎么优化？ ★★★ 生产题
【参考答案】

按 RAG 流水线六个环节系统优化：

① 切分优化：

  • Semantic Chunking：按语义切，而不是固定字符数。
  • Parent-Child 切分：小块精确匹配、返回大块给更多上下文。
  • 重叠切分：chunk_overlap 150-200 避免边界信息丢失。

② 嵌入优化：

  • 选对模型（中文优先 BGE、M3E、text-embedding-3-large）。
  • 针对领域数据微调 Embedding（sentence-transformers）。

③ 检索策略：

  • 混合检索：向量检索（语义）+ BM25（关键词）融合。
  • HyDE：先让 LLM 生成假设答案，用答案做检索（比用问题更相似）。
  • Query Rewriting：用 LLM 改写查询，拆成多个子查询。

④ 重排序：

  • 取 top-20 用 Cross-Encoder（BGE-Reranker / Cohere Rerank）精排成 top-5。
  • 重排比粗排的语义理解更深，显著提升准确率。

⑤ 生成控制：

  • Prompt 约束："根据以下上下文回答，如果不知道就说不知道"。
  • 引用标注：强制模型引用 [1] [2] 溯源。

⑥ 架构升级：

  • Agentic RAG：让 Agent 判断要不要检索、检索什么、检索几次。
  • GraphRAG：知识图谱结构化知识，支持多跳推理。
  • Self-RAG：模型自主决定检索时机。
Q176 "Lost in the Middle" 问题是什么？怎么解决？ ★★★ 2026 前沿
【参考答案】

现象：LLM 对长上下文的开头和结尾关注度高，中间内容容易被忽略，即使关键信息在中间也可能被漏掉。

解决方案：

  • Reranking：把最相关的内容放在上下文开头或结尾。
  • 压缩上下文：用 LLMLingua 等工具压缩无关部分。
  • 结构化提示：用标题、序号明确区分不同部分。
  • 限制上下文数量：只塞 top 3-5，不是塞越多越好。
  • Long-Context 优化模型：Claude、Gemini 的长上下文表现更好。
Q177 如何评估 RAG 系统的效果？ ★★
【参考答案】

分检索和生成两阶段评估：

检索阶段指标：

  • Precision@K：top-K 中相关文档的比例。
  • Recall@K：top-K 覆盖了多少真实相关文档。
  • MRR（Mean Reciprocal Rank）：第一个相关结果的排名倒数。
  • NDCG：考虑排序质量的综合指标。

生成阶段指标：

  • Faithfulness（忠实度）：答案是否基于检索内容，没有幻觉。
  • Answer Relevancy：答案是否真的回答了问题。
  • Context Precision：提供的上下文中相关部分的占比。

主流工具：RAGAS、TruLens、LangSmith Evaluations，都支持 LLM-as-a-Judge。

Q179 向量检索的原理？HNSW 算法怎么工作？ ★★★
【参考答案】

向量检索本质：找最近邻（Nearest Neighbor）。暴力搜索 O(n) 在大数据下不可行，所以用近似算法 ANN。

HNSW（Hierarchical Navigable Small World）是最主流的算法：

  • 构建多层图：底层最密集（全部节点），上层稀疏。
  • 每个节点连接若干邻居（基于邻近性）。
  • 搜索时从顶层开始，每层找局部最近，逐层下降精细化。
  • 时间复杂度 O(log n)，在百万级数据下几十毫秒返回。

其他常见算法：IVF（聚类分桶）、LSH（哈希）、PQ（乘积量化，节省内存）。

点积方向 + 长度都考虑向量已归一化时等同余弦

Q182 LangChain 有什么劣势？如果让你设计 Agent 框架你会怎么优化？ ★★★ 深度追问
【参考答案】

诚实说 LangChain 的问题（面试真实度加分）：

  • 过度抽象：太多 Chain/Agent 封装层，debug 困难，堆栈难追。
  • 性能开销：简单任务被包装成复杂对象，额外开销不小。
  • 版本动荡：v1.0 之前 API 频繁 breaking change，升级痛苦。
  • 依赖沉重：安装一个 Chain 可能拉一堆底层依赖。
  • 可观测性不足：需要额外接 LangSmith 才能看清执行过程。

自己设计会优化这些点：

  • 用状态图 / DAG 代替隐式 Chain，节点边清晰可视化（LangGraph 思路）。
  • 原生支持流式 + 异步 + 取消，不是靠 adapter。
  • 工具层做薄，用 OpenAI Function Calling 标准，而不是自定义 Tool 抽象。
  • 内置 tracing（OpenTelemetry），每步有 trace_id 方便 debug。
  • 把 Memory / Retriever / Guardrails 做成独立中间件，解耦。
  • 插件化 LLM 适配层，方便切换模型。
Q183 MCP（Model Context Protocol）是什么？ ★★★ 2026 热点
【参考答案】

MCP 是 Anthropic 2024 年提出的开放协议，2026 年已成为 AI 应用接入外部工具的行业标准。类比 AI 界的 "USB-C"。

核心角色：

  • MCP Server：把你的数据源/工具/API 暴露成 MCP 服务。
  • MCP Client：AI 应用（Claude Desktop、Cursor、Claude Code 等）按 MCP 协议调用。
  • 传输层：STDIO 本地 或 SSE/HTTP 远程。

为什么重要：

  • 解决 M×N 问题：M 个 AI 客户端 × N 个工具 → 过去要写 M×N 个集成，MCP 后只要 M+N 个。
  • 工具即插即用：写一个 MCP Server，所有支持 MCP 的 AI 都能用。
  • 生态爆发：2025-2026 MCP Server 数量从 100+ 增长到 3000+，涵盖数据库、API、文件系统、浏览器等。

和 Function Calling 的区别：

  • Function Calling 是 LLM 调用工具的机制（模型内部能力）。
  • MCP 是工具暴露给 LLM 的标准协议（外部接口约定）。
  • 两者配合使用：MCP Server 提供工具，LLM 通过 Function Calling 调用。
Q184 A2A（Agent-to-Agent）协议了解吗？ ★★
【参考答案】

A2A 是 Google 2025 年提出的 Agent 间通信协议，解决不同厂商的 Agent 如何协作问题。

核心概念：

  • Agent Card：每个 Agent 有描述文件，说明能力、接口、认证方式。
  • 任务交接：一个 Agent 可以发现其他 Agent 并委托任务。
  • 跨生态：理论上 LangGraph Agent 可以调用 CrewAI Agent。

和 MCP 区别：MCP 是 Agent → Tool；A2A 是 Agent → Agent。

当前状态：2026 年还在早期，大厂在推，实际落地还不多。了解概念即可，面试能说出就是加分。

Q185 AG-UI 协议是什么？ ★★ 前端相关
【参考答案】

AG-UI（Agent-User Interface）协议：标准化 Agent 和前端 UI 的交互。

解决的问题：Agent 流式输出、工具调用状态、中间推理过程怎么展示给用户。

核心特性：

对你这个前端转 AI Agent 方向的人，AG-UI 是最有亲和力的切入点 —— 你的前端经验能直接发挥。

Q186 Agent 应用的成本怎么控制？ ★★ 落地题
【参考答案】

Agent 应用成本主要是 token 费用，6 个优化方向：

  • 模型分级：简单任务用小模型（Claude Haiku / GPT-4o-mini），难任务才上大模型。
  • Prompt 精简：去掉无用说明，删掉重复 few-shot 例子。
  • Context 压缩：历史对话定期 summarize，只保留要点。
  • Semantic Caching：相似问题命中缓存（Redis + 向量检索）。
  • Tool Result 截断：工具返回只保留核心字段，别把整个 JSON 塞给 LLM。
  • Batch 处理：能批量的就批量，利用 Anthropic Batch API / OpenAI Batch（50% 折扣）。
💡 记忆技巧：实战经验：一个多 Agent 客服系统，上线时平均 $0.35/对话，优化后降到 $0.08/对话，关键就是分级 + 缓存。
Q187 怎么给 Agent 加护栏（Guardrails）保证安全？ ★★
【参考答案】

护栏分输入侧和输出侧：

输入侧：

  • Prompt 注入检测：识别 "忽略之前的指令" 这类越狱尝试。
  • 敏感信息脱敏：身份证、手机号、信用卡等入库前掩码。
  • 有害内容过滤：Moderation API 检测暴力 / 色情 / 仇恨内容。
  • 业务规则校验：参数边界、白名单校验。

输出侧：

  • 事实一致性：RAG 场景检查答案是否基于上下文。
  • 格式校验：JSON Schema、正则，不符合让 LLM 重新生成。
  • 敏感话题屏蔽：政治、宗教、涉及未成年人的内容。
  • PII 再次扫描：LLM 输出的内容再扫一遍，防止泄漏。

工具：NeMo Guardrails（NVIDIA）、Guardrails AI、LLaMA Guard。

Q188 Agent 应用的可观测性怎么做？ ★★
【参考答案】

Observability 核心是 "每次执行能回放、能 debug、能分析"。

  • Tracing：每次请求一个 trace_id，串联 LLM 调用、工具调用、子 Agent 调用。
  • 关键指标：首 token 时间、总响应时间、工具调用次数、成本、成功率。
  • 错误追踪：LLM 报错、工具报错、解析失败都要有堆栈和上下文。
  • 流量回放：线上 bad case 可以离线复现。
  • 数据飞轮：把用户真实问题回流到评估集，持续迭代。

工具：LangSmith（LangChain 出品）、Langfuse（开源）、Helicone、Arize Phoenix、OpenTelemetry。

Q189 给你个项目 —— 为企业做一个客服 Agent，你怎么设计？ ★★★ 系统设计题
【参考答案】

这是终面/架构面常见的设计题，按 5 层架构讲：

① 数据层：

  • 知识库：产品文档、FAQ、历史工单切分入向量库（Qdrant）。
  • 业务系统：订单、用户信息通过 API 工具查询，不入向量库。

  • 工具：订单查询、退款 API、人工转接、物流查询等。

④ 接入层：

  • 兜底：关键操作（退款、改订单）必须二次确认 + 人工审核。
  • 护栏：输入侧防注入，输出侧防 PII 泄漏。
💡 记忆技巧：成功指标：自动化解决率、平均对话成本、用户满意度、首次解决率。能说出这些让面试官觉得你真落地过。

10.1 艾宾浩斯遗忘曲线原理

德国心理学家艾宾浩斯发现：新记忆的知识会快速遗忘，但如果在特定时间点复习，遗忘速度会显著减慢。关键时间点是：

时间点不复习的遗忘率复习后保留率
20 分钟后42%100%
1 小时后56%95%
1 天后74%90%
2 天后78%80%
7 天后77%70%
30 天后79%60%

所以最优复习节奏是：

  • 第 1 天：学习（首次）
  • 第 2 天：复习一次
  • 第 4 天：复习一次
  • 第 7 天：复习一次
  • 第 15 天：复习一次
  • 第 30 天：复习一次

6 轮复习后，大部分内容会进入长期记忆，面试时能自然脱口而出。

10.6 心态建议
